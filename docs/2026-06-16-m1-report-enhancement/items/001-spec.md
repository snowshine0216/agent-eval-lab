## 1. The problem

`M1-final-report.md` is auto-generated by `reports/m1.py` and aggregates
everything to **per-(condition, domain)** granularity: pass^k + CI, macro
composite, Pareto (3 axes), planned comparisons, failure-taxonomy **counts**,
validity. That is an outline. It cannot answer the questions an analyst actually
asks:

- *Where did the cost / rounds / tokens go, per condition and per task?* — today
  cost and rounds appear only as a single number buried inside the Pareto axis
  tables; tokens are an order-of-magnitude figure; there is no prompt/completion
  split, no spread, no `$` rollup, no tool-call or stop-reason view.
- *Why did a model fail a task?* — today only a category tally
  (`agent_failure / oracle_red ×13`). No per-task breakdown, no "which oracle
  test failed", no "what the model edited".
- *How do D and B fail?* — there is **no** per-set deep dive for any domain but
  F, and the F one ([M1-F-failure-analysis.md](../../../reports/agentic-v1/M1-F-failure-analysis.md))
  is hand-written, F-only, and not reproduced by the next `report-m1` run.

## 2. Decisions (owner-confirmed 2026-06-15)

1. **Reproducible generator**, not a one-off doc — every `report-m1` run
   regenerates the full quantitative depth. (The harness is deterministic and
   spec-hashed; a hand-written report rots on the next run.)
2. **Overview + per-domain subreports** — `M1-final-report.md` is the thin
   cross-domain rollup; the depth lives in `M1-<domain>-report.md`, one per
   domain that has runs.
3. **Auto subreport coexists with the hand companions** — the generator never
   clobbers prose. `M1-F-failure-analysis.md` and `M1-F-report-NOTES.md` remain
   hand-authored; the overview links to both the auto subreport and the
   companions. The auto subreport supplies everything quantitative; the
   **task-defect candidates** + **shared failing oracle unit(s)** (§6) are the
   closest auto-derivable proxy for the hand-written "shared root cause" prose.

**Hard boundary — what a deterministic generator will NOT produce:** free-form
root-cause synthesis ("every model read the contract as a synchronous check").
That stays in the hand companions. The auto subreport instead renders the
*evidence* that motivates such prose (failing test names, shared-gap
intersection, edited files), so the report is self-explanatory without
inventing narrative.

## 3. What the records already contain (feasibility — verified)

Every run JSONL record (`runs-m1-<slug>-<domain>.jsonl` / `runs-dset-<slug>.jsonl`)
carries, per attempt:

- `trajectory.usage = {prompt_tokens, completion_tokens, latency_s}`
- `trajectory.rounds`, `trajectory.tool_call_counts = {tool: n}`,
  `trajectory.stop_reason`, `trajectory.safety_cap_bound`,
  `trajectory.max_rounds`, `trajectory.safety_cap`
- `trajectory.final_state = {repo, candidate_base_sha, target_paths, files}`
  — the model's actual edits and the declared in-scope paths
- `grade = {grader_id, passed, score, evidence}` where **evidence is
  grader-specific**:
  - **F** (`grader_id = all_of` → `sub_results[].grader_id = node_execution`):
    sub-evidence `{execution, status, exit_code, counts, tests:[[name, "passed"|"failed"], …], stdout, stderr, execution_hash, displaced_paths}`
    — a **per-oracle-test pass/fail list** plus `displaced_paths` (the
    **oracle-overlay collision** set, glossary "displaced path" — *not*
    out-of-scope edits, which are trajectory-derived; §5/§5a).
  - **D** (`grader_id = fact_key`): evidence
    `{level, required_not_on_page, missing_required, present_forbidden, page_snapshot_sha256}`
    — fact-level pass/miss.
  - **Administrative** (glm F3): `grade.evidence.marked_failed_not_executed = true`,
    0 rounds / 0 tokens / empty turns.

So the entire **quantitative** content of the hand-written F analysis is
auto-derivable. The one gap (per-test detail shape) differs by grader → handled
by a grader-aware adapter (§5).

## 4. Architecture & data flow

Two new **pure** report-layer modules (build + render, no I/O — mirrors the
existing `m1.py` / `final.py` split), plus a CLI wiring change. Nothing else moves.

```
report-m1 (cli.py)
  ├─ build_m1_report(...)            # unchanged — overview data
  ├─ build_m1_detail(...)            # NEW: per-domain detail data (reports/m1_detail.py)
  │     ├─ evidence_gap(grade)       # NEW: grader-aware adapter over GRADE (reports/evidence_summary.py)
  │     ├─ edit_paths(trajectory)    # NEW: edited/out-of-scope paths over TRAJECTORY (reports/edit_paths.py)
  │     ├─ task_defect_candidates()  # shared unanimous-fail predicate (reports/defects.py — extracted from final.py)
  │     └─ classify_run(run)         # existing fc-v4 classifier, now per task×condition
  ├─ render_markdown(report)         # m1.py — overview, + new efficiency rollup + subreport links
  └─ render_detail(detail, domain)   # NEW: one markdown subreport per domain (reports/m1_detail.py)
```

Two distinct derivation sources, deliberately separated: `evidence_gap` reads
the **`GradeResult`** (oracle units, `displaced_paths`); `edit_paths` reads the
**`Trajectory`** (the agent's `str_replace`/`write_file` tool-call targets). They
must not be merged — displaced paths and out-of-scope edits are different
concepts (CONTEXT.md), and they live on different records.

- **Inputs:** the *same* `outcomes_by_condition_domain` already handed to
  `build_m1_report`, the `ExperimentSpec`, and the `PricingSnapshot`. No new
  data source; no second pass over disk.
- **Determinism:** every derived value is a pure function of records + spec +
  pricing; all iteration is over sorted keys (conditions, domains, task_ids).
  A determinism test asserts byte-identical output for a fixed input (mirrors
  `tests/experiments/test_m1_determinism.py`).
- **`m1.py` size:** already 473 lines; the detail logic goes in its own module
  rather than growing it further (project rule: a file that does too much gets
  split).
- **Shared `reports/defects.py`:** extract `TaskDefectCandidate` and the
  unanimous-fail predicate (currently private in `final.py`:
  `_task_defect_candidates`) into a pure shared module; refactor `final.py` to
  import it (no behavior change — guarded by its existing tests) and have the M1
  subreport call it per domain. One glossary-critical definition, not two
  (DRY). Trade-off: this touches the v1 `final.py`; the refactor is a pure
  extract-and-import, low-risk, covered by `final.py`'s tests.

## 5. `reports/evidence_summary.py` — grader-aware gap adapter (the key abstraction)

A single pure entry point maps one `GradeResult` to a small, render-ready value:

```python
@dataclass(frozen=True, kw_only=True)
class EvidenceGap:
    grader_id: str
    oracle_total: int | None        # F: len(tests); D: None (no denominator in grade — Q6)
    oracle_passed: int | None       # F: #tests passed; D: None
    failing_units: tuple[str, ...]  # F: failing test names; D: missing_required + present_forbidden
    displaced_paths: tuple[str, ...]  # oracle-overlay collisions (F node_execution); else ()
    administrative: bool            # marked_failed_not_executed
    status: str                     # "passed" | "failed" | "incomplete" | "not_executed" | "no_answer"

def evidence_gap(grade: GradeResult) -> EvidenceGap: ...   # reads GradeResult ONLY (Q6)
```

- **Grade-only contract (Q6):** never reads the `VerificationSpec`. F yields a
  clean `passed X/Y tests` from the `tests` list; D yields the *named missed
  facts* with `oracle_total = None` (the denominator lives on the spec, not the
  grade — the F/D asymmetry is inherent and reported honestly).
- **Degraded evidence branch:** a `fact_key` failure can carry
  `{"error": "no assistant message in trajectory"}` (model produced no answer) —
  the adapter maps it to `status="no_answer"`, `failing_units=()`, never raises.

- Dispatch on `grade.grader_id` (and the leaf `sub_results[].grader_id` for
  `all_of`). Add a branch per grader; an unknown grader yields a minimal gap
  (`status` from `grade.passed`, no oracle detail) — never raises.
- This is the **only** place that knows evidence internals, so adding the B
  grader later is one new branch + its test. Keeps the renderer grader-agnostic.
- `displaced_paths` carries the **glossary "displaced path"** signal verbatim
  (oracle-overlay collision) — **not** out-of-scope edits, which are a separate,
  trajectory-derived concept (§5a).

### 5a. `reports/edit_paths.py` — trajectory-derived edit signals

```python
@dataclass(frozen=True, kw_only=True)
class EditPaths:
    edited: tuple[str, ...]         # paths targeted by str_replace / write_file tool calls
    out_of_scope: tuple[str, ...]   # edited − target_paths (CONTEXT.md "out-of-scope edit")

def edit_paths(trajectory: Trajectory, *, target_paths: Sequence[str]) -> EditPaths: ...
```

- Reads **only** the `Trajectory` (the agent's edit-tool-call targets) plus the
  task's declared `target_paths`. **Descriptive, never a verdict** — F has no
  `OnlyModifies` leg, so an out-of-scope edit is reported, never auto-failed.
- Edit-tool name set is grader-world-aware (code-world: `str_replace`,
  `write_file`); an unknown tool contributes no edited path (fail-quiet).

## 6. Per-domain subreport `M1-<domain>-report.md` — content (auto, per domain with runs)

Sections, in order — every value derived, no prose:

1. **Header** — domain, conditions present, `k`, task count, grader/oracle type,
   pinned repo + base SHA (F) / docs host snapshot note (D).
2. **Task quick-reference** — per task: `target_paths` (files to edit),
   grader/oracle type, and oracle-test count where the grade carries one
   (`EvidenceGap.oracle_total`; F only — D has no grade-side denominator, Q6).
3. **Cross-model summary** — task × condition matrix: pass^k contribution
   (passed/total valid trials) + the per-trial `✅❌✅❌✅` string + the
   dominant `stop_reason`.
4. **Per-task detail blocks** — for each task, per condition:
   - pass matrix (per-trial), rounds (median [min–max]), tokens
     (prompt / completion / total), `cost_usd`, tool-call counts, safety-cap
     hits, stop-reason.
   - **grader-aware gap** from `EvidenceGap`: F → "passed X/Y oracle tests;
     failing: `<name>`, …"; D → `missing_required` / `present_forbidden` facts.
   - **edit signals**: files edited (`EditPaths.edited`), of which **out-of-scope**
     (`EditPaths.out_of_scope`, distinct from the oracle-overlay
     `EvidenceGap.displaced_paths`). Both labeled with their canonical names.
5. **Task-defect candidates** (canonical — glossary, ADR-0013) — tasks that
   *every* non-blocked condition with records unanimously fails. Reuses the
   shared `reports/defects.py` predicate (§4). Flagged **for human review, never
   auto-classified** ("unanimity defaults to *hard, not defective*"). This is the
   review-queue the M1 report currently lacks (F1/F2 surface here).
   - **Supporting detail (not a named concept):** beneath each flag, the
     **shared failing oracle unit(s)** — the intersection of
     `EvidenceGap.failing_units` across the failing conditions ("all failing
     conditions miss `<test>`"). Descriptive color that sharpens *why* a task is
     hard; also rendered for non-unanimous tasks. Empty intersection ⇒ "divergent
     failures (no shared unit)". (This replaces the earlier coined "convergence
     signal".)
6. **Per-condition efficiency rollup** (domain-scoped) — the same columns as the
   overview rollup (§7), one row per condition.
7. **Failure classification per task × condition** (fc-v4) — re-uses
   `classify_run`; shows the category at task granularity, not just a domain tally.

**Edge cases the renderer must handle explicitly (not silently drop):**

- **Void / incomplete task** (`< k` valid trials): shown with `status =
  incomplete`, excluded from pass^k, flagged — consistent with D34.
- **Administrative trial** (`marked_failed_not_executed`): rendered as
  "administrative 0/k — not executed (owner decision)", **never** as a real
  0-round/0-token failure. `EvidenceGap.administrative` drives this.
- **Invalid (env-masked) trial:** read from `ReplacementOutcome.attempts` (the
  full attempt history; `valid_runs` is the k-valid subset), counted and labeled
  separately from model failures (the SiliconFlow 403/429 case) so infra noise
  is not read as a gap.
- **Condition missing a task entirely:** explicit "—" row, not an absent row.

## 7. Overview `M1-final-report.md` — additions (existing sections unchanged)

- **NEW "Efficiency & cost" rollup** — per (condition, domain): rounds median
  [min–max], prompt tok, completion tok, total tok, `cost_usd`, total tool
  calls, safety-cap hits, dominant stop_reason. (Reuses a new richer efficiency
  builder; see §8 — the existing `EfficiencySummary` exposes only
  `median_rounds` + `total_tokens`, too thin for this.)
- **NEW "Subreports" block** — links to each generated `M1-<domain>-report.md`
  and to the hand companions (`M1-F-failure-analysis.md`, `M1-F-report-NOTES.md`).
- **NEW deterministic one-line headline per domain** — "best pass^k: `<cond>`
  (x.xxx); cheapest on cost-frontier: `<cond>`" — computed from existing
  per-domain results + Pareto frontier, no prose.
- **Rename the existing "Failure taxonomy" heading → "Failure classification
  (fc-v4) per condition"** (`m1.py:374`). The glossary's `RunClassification`
  entry explicitly *avoids* "failure taxonomy" (reserved for `FailureCategory`);
  `final.py` already uses the correct "Failure classification" heading, so this
  aligns the M1 report with both. One-line change; update the assertion in
  `tests/reports/test_m1_render.py:70` to require the new heading. (`validation.py`
  carries the same mislabel — spun out to a separate follow-up, not this spec.)
- Everything else (per-domain scores, composite, Pareto, comparisons,
  validity) stays byte-for-byte as today.

## 8. Efficiency builder (shared by §6.6 and §7)

Add a richer per-(condition, domain) efficiency value in the report layer
(either extend `EfficiencySummary` with additive optional fields, or a new
`reports`-local `CondDomainEfficiency` — **recommend the latter** to avoid
touching the Pareto consumers of `EfficiencySummary`):

```
# time-to-completion — RIGHT-CENSORED for budget-capped runs (glossary: censoring)
rounds_median, rounds_min, rounds_max,
censored_count: int,                  # runs in the group that are budget-capped
cap_bound: int | None,                # the cap the censored runs hit (max_rounds / safety_cap)
# resource — OBSERVED, summed over ALL valid runs incl. capped (never censored)
prompt_tokens, completion_tokens, total_tokens,
cost_usd | None,                      # condition_cost_usd over the task's valid runs
tool_call_totals: Mapping[str, int],
safety_cap_hits: int,                 # stop_reason == "safety_cap" (backstop)
max_rounds_hits: int,                 # stop_reason == "max_rounds" (primary budget, F-ablation)
stop_reason_counts: Mapping[str, int]
```

Computed over the **valid** runs for that (condition, domain) — and re-callable
over a **single task's** valid runs for the per-task blocks. `cost_usd` reuses
`experiments.pricing.condition_cost_usd(runs, cond, pricing)` on the run subset;
`None` when the condition has no price (consistent with the overview today).

**Censoring discipline (glossary `censoring`, §D.3):** the renderer marks the
rounds cell whenever `censored_count > 0` — e.g. `16 (2/5 right-censored at cap
20)` — so a median mixing capped and completed runs is never read as a
completion time. A run is censored iff `stop_reason ∈ {safety_cap, max_rounds}`
(equivalently `safety_cap_bound` / `max_rounds_bound`). Tokens and cost are
**observed** and summed over all valid runs including capped ones — never
annotated as censored.

## 9. CLI change — `report-m1`

- After writing the overview to `--out`, write `M1-<domain>-report.md` for each
  domain present, into the **same directory as `--out`** (so the existing
  `gen-final-report.sh` keeps working with no path edits).
- **Filename note:** `M1-F-report.md` already exists as the older *thin*
  single-domain auto report. The new richer subreport **reuses that filename and
  supersedes it** (same generator, more content). The hand companions
  (`M1-F-failure-analysis.md`, `M1-F-report-NOTES.md`) keep their distinct names
  and are untouched.
- Add `--subreports / --no-subreports` (default **on**) and an optional
  `--subreport-dir` override.
- The overview's "Subreports" links are computed from the same set, so links and
  files never drift.

## 11. Out of scope

- Re-running any model; changing `grade.passed`, pass^k, CI, or comparison math.
- Authoring root-cause prose (stays in hand companions).
- The B domain's grader adapter — added as one `evidence_gap` branch + test when
  B runs exist; the generic path renders B efficiency meanwhile.
- HTML / charts — markdown only, matching the rest of `reports/`.

## 12. Risks & mitigations

- **Subreport size growth** with many conditions × tasks → acceptable; one file
  per domain, and per-task blocks are bounded by task count (F=3, D≈15/25).
- **Grader drift** (a future grader changes evidence shape) → isolated to
  `evidence_summary.py`; unknown-grader path degrades gracefully, covered by a
  test.
- ~~`ReplacementOutcome` access to invalid/administrative trials~~ **(resolved
  during grilling):** invalid trials are on `ReplacementOutcome.attempts`;
  administrative trials are in `valid_runs` flagged by
  `grade.evidence.marked_failed_not_executed`. Both reachable from the outcomes
  map already passed in — no second data source, no plan-time unknown.
